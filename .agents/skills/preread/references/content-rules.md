# 中身の生成規則

モック検証(poc-paper, v1〜v16)で確定した要件の集約。

## 書き直し

- 内容の範囲を維持する: 易しくしても、原文が主張していないことを足さない。
  主張の強度(「示した」と「示唆した」の区別)を変えない
- 読者が持っていない前提(読者プロファイルとの突き合わせで特定)は、
  本文の流れの中で穴埋めする。前提解説を別章に隔離しない
- 略語は初出で正式名称を展開する
- 用語には `.term` ポップオーバーを付け、説明に英語の原語を併記する
- 数式は `.formula` ブロックに入れ、「数式を日本語で読む」開閉で
  日本語訳(記号を言葉に置き換えた読み下し)を添える
- 受け身で仕組みを隠さない: 「誰が・どのように・何を」を明示する
  (「収集される」ではなく「データプロキシが呼び出しの境界で軌跡を記録する」)

## 構造の図解化

- 散文で説明できる構造(ループ・対応関係・時系列・段階)は可能な限り
  図解に変換する
- 原典の重要な図は「日本語で読む」形式で扱う: 図を再描画し、
  段送りボタン(FIG_STEPS)で見る順序ごとに該当部をハイライトしながら案内する
- 誤った可視化は誤った直感を植え付ける。数式の自動可視化はしない

## 心像図(ミニマップ)

- 独立ビューにせず本文と一体化する(コードエディタのミニマップの概念)。
  地図のノード → クリック → 要約 → 本文へジャンプ、の動線
- 骨格は文書タイプで切り替える(doc-types.md 参照)
- 生成後、原文と突き合わせて構造の誤りがないか確認する

## 埋め込みクイズ

- セクションごとに置き、能動的な読みを保つ。正誤判定と解説(`.kaisetsu`)は
  あらかじめ埋め込む
- 選択肢は長さを揃える(書式から答えが漏れないように)
- 「本文を読んでいれば答えられるが、読み飛ばすと間違える」水準にする

## 現実世界で回してみる

- ワークトエグザンプル形式(追う)単独。意思決定シナリオや操作型説明は
  比較検証の結果、不採用
- 題材は読者プロファイルの現場に寄せる。理論の登場人物を読者の
  日常の具体物(担当サービス、使っているツール)に置き換えて 1 周追う
- 段階があるなら段送りコンポーネント(#sim-steps)で 1 段ずつ見せる

## 用語集

- 教材末尾に置く。`.gitem` に用語(日本語)・原語(英語)・短い説明

## 結びの判定ブロック(.verdict)

- 3 段落構成を守る:
  1. この教材で掴めたこと(骨組み・限界・現場への当てはめまで)と、
     原典でしか確かめられないこと(主張の正確な強度・全文・実装詳細)
  2. **進むなら**: どの章の知識を念頭に、原典のどこから読むか + 原典リンク
  3. **止めるなら**: それも正しい判断だと明示し、持ち帰る問いを 1 つ添える
- 構想中の機能や外部コマンドを指す結びにしない。教材はここで完結する

## メタ情報

- hero の kicker は「文書種別 ─ 読了目安 N 分」。種別を偽らない
  (実証なしの論文は「立場表明」と書く)
- .meta に著者・分野・種別。種別には検証の程度を括弧で添える
- フッターに「LLM による再構成であり、正確さは原典で確認してください」を残す
