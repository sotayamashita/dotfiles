# Ja Prose Rigor

日本語の原稿の論証を締める。扱うのは主張が妥当か・論証が反論に耐えるか(深層)で、文が読みやすいか(表層)は ja-prose-clarity が扱う。読みやすさの規則(受身、漢語、冗長、まとめ行、見出し、LLM 口調)はそちらにあるので、ここでは繰り返さない。

## 使い方

書き上げてから点検するスキル。まず ja-prose-clarity で文を軽くし、次にこのスキルで論証を締める。論証の厳密さは、書いたあとに次のチェックリストを写して1項目ずつ確認する。

```
論証の点検:
- [ ] 推量を機械的に断定へ変えていないか(根拠で確定した命題だけ断定)
- [ ] 異なるものを同じ語でくくっていないか
- [ ] 複数の要因を単一の原因に還元していないか
- [ ] 章・節をまたいで概念の扱いが一致しているか
- [ ] 因果を主張した箇所に、その機構が一文あるか
- [ ] 「必ず」を条件付きに直したか
- [ ] 挙げた例が主張の全体を支えているか
- [ ] 「次節で扱う」と逃がした論点を回収したか
- [ ] 逆接(「ただし」)で終えて宙吊りにしていないか
```

## 段落と論証の構成

パラグラフライティングを基本とする。段落は論証の一歩で、読者は段落単位で論理を追う。詳細と例は [references/composition.md](references/composition.md)。

1. 一段落に一トピック。場面の進行(調査、報告、検証、評価)が混ざった長い段落は一歩ずつに分ける。
2. 段落の最初の文で、その段落が何の話かわかるようにする。
3. 段落の先頭で、前の段落との論理関係を接続表現で示す(「であれば」「実際」「しかし」)。
4. 論証は一方向に進める。結論を出してから反論を処理して言い直さない。反論と疑念を処理してから結論を一度だけ置く。
5. 例への弁明(作為的に見える等)は山場の直後に挟まず、次の節の冒頭でまとめる。
6. 読者が立てそうな誤解は、明示的に否定してから本当の理由を述べる。
7. 山場で効かせたい情報(数値、固有の事実)を、手前の段落で先出ししない。
8. 否定・限定するときは、否定する命題を「」で正確に書き出す。漠然とした否定で済ませない。
9. 前方参照(「後の章で扱う」)は論証が一段落した位置(段落末・節末)に置く。

## 論証の厳密さ

文章の論理にツッコミどころを残さない。詳細と Bad/Good は [references/argument-checks.md](references/argument-checks.md)。

10. 推量・可能性・読者の疑念・反実仮想を機械的に断定へ変えない。断定に直せるのは本文内の根拠で命題が確定している場合だけ(ja-prose-clarity 規則15と対。あちらは根拠なきぼかしを消す)。
11. 区別すべき対象(別々の決定、別々の原因)を一括りの語でくくらない。
12. 複数の要因がある事象を単一の原因に還元しない。
13. 章・節をまたいで、同じ概念の分類・定義・地位を一致させる。
14. 因果を主張するときは、その機構(なぜそうなるか)を一文で示す。
15. 検出・保証・解決を「必ず」できるように書かない。条件付きで述べる(「〜しやすい」「〜が成り立つときに限り」)。
16. 挙げた例が主張の全体を支えているか確認する。支えないなら主張を例に合わせて狭める。
17. 前方に逃がした論点は本当に回収する。回収しない伏線を張らない。
18. 譲歩・限定(「ただし」「とはいえ」)を置いたら、その後で必ず論を進める。
19. 節の中心語は、その節以前に定義・対象範囲を述べてから使う。
20. 複数の概念を上位語にまとめるときは、命名の直前に、それらが同じものに帰着すると一文で述べる。

## 演出の抑制

修辞は効果を生む箇所だけで使う。全面禁止ではなく節度の規範。詳細は [references/composition.md](references/composition.md)。

21. 溜め(「ここには〜が潜んでいる」)や修辞疑問は、緊張が議論に効く要所に限る。
22. 短い決め台詞を独立段落にして緊張を作る演出を多用しない。段落内の短い体言止めは山場に限る。
23. 命令調の断定(「〜してはならない」)より、作業者の判断として書く形(「〜するわけにはいかない」)を選ぶ。
24. 転回点を過剰に劇的にしない。事実を述べる一文で足りることが多い。
25. 帰結の列挙で事故や危険を煽らない。

## 語りと誠実さ

論証の語り手としての規範。文の読みやすさに属する語り(受身、具体語、術語の一貫)は ja-prose-clarity。

26. 論証の中で読者を「あなた」と呼ばず役割名(「開発者」「読者」)で書く。二人称は場面導入(「〜としよう」)や結びなど要所に限る。
27. 「入社2年目のエンジニアが」のような架空の人物設定を無意味に冠しない。
28. 例が作為的に見えうるなら隠さず、読者の疑念を先回りして認め、現実にあり得る根拠を短く添える。その根拠は著者の断定ではなく、読者の経験に訴える一般的事実や通説に求める。
29. 確認していないことを、確認したかのように滑らかに書かない。
