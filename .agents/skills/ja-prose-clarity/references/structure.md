# 構造レベルの書き換え規則

前後の文脈がないと判定できない問題。番号は SKILL.md の規則番号と同じ。

## 21. 既知→未知の順序違反 → 説明を先、詳細を後

読み手がまだ知らない詳細・列挙を先に出さない。箇条書きの前に、それが何の列挙かを一文で書く。導入文は「構成は次の3つ:」のように名詞で終える(述語+コロンは規則19に当たる)。

- Bad: 箇条書きを先に出し、その説明を後に置く
- Good: 「分け方は次の2つ:」と名詞終わりの一文で導入してから箇条書きを出す

## 22. 命題密度が高すぎる → 1段落に主張は1つ

個々の文は読めるのに全体が重いとき、段落に主張が詰まりすぎている。段落を増やし、1段落を1つの主張とその根拠だけにする。

- Bad: 3段落に14の主張
- Good: 1段落1主張に分割して段落数を増やす

## 23. 強調位置の違反 → 文末に核心を置く

文末は読み手が最も重く受け取る位置。書き終えたら最後の語句を見て、そこにあるのが譲歩や但し書きなら核心と入れ替える。段落の最後の文も同じ。規則21(文頭は既知)と対で、文の両端を設計する。

- Bad: 「この設計で応答時間は半分になりますが、注意点もいくつかあります」
- Good: 「注意点はいくつかあるものの、この設計で応答時間は半分になります」

## 24. 話題の不連続 → 段落内で話題を揃える

段落の話題を一つ決め、各文の文頭をその話題の系列で揃える。新しい話題は段落を分けるか、前の文の文末で予告する。

- Bad: 「このAPIは高速だ。開発者の生産性は設計に依存する。キャッシュ層が応答を返す。」(文ごとに話題が変わる)
- Good: 「このAPIは高速だ。キャッシュ層が大半の応答を返すからだ。この設計は開発者の生産性も上げる。」(話題が繋がる)

## 25. 偽の結束 → 論理関係があるときだけ接続詞を置く

文頭の接続詞を消して意味が変わらないなら消す。並列の「また」が続くなら箇条書きにする。関係を示すなら関係の中身を書く(「その結果」「その代わり」)。同じ接続詞を連続させない。逆に、前の段落との論理関係が実在するなら、段落の先頭で接続表現によって明示する(「であれば」「実際」「しかし」)。

- Bad: 「また、〜です。さらに、〜です。また、〜です。」(並列の水増し)

## 26. 冗長なフレームマーカー → まとめ行と自己言及を消す

箇条書きやセクションの後に「つまり〜」の一行や、毎回同じ形式の要約行を置かない。必要なまとめは最後の項目の末尾か、次の段落の冒頭に移す。文書自身への言及(「本 ADR では」「本稿では」)も消し、定義や範囲は本文で直接書く。

- Bad: 箇条書き3項目の後に「つまり導入はベンダースワップであり、組織変革ではない。」が浮く
- Bad: 毎セクション末尾に「優位性: 太字まとめ」を置く
- Bad: 想像上の読者との問答や、メタな枠取り(「〜という発想である」)で主張を紹介する
- Bad: 「本書もそれを否定しない」のような著者の立場の弁明

## 27. 情報量のない見出し → 内容を特定する句にする

見出しは、その節が答える問い、または扱う対象を指す具体的な句にする。種別だけの見出し、作業手順だけの見出し、節の結論を言い切ってオチを先に明かすセリフ見出しの3方向を避ける。疑問形か名詞句かは本文のトーンに合う方でよい。

- Bad: 「補足」「例に戻す」(何の話か特定できない)
- Bad: 節の結論をそのまま見出しにして、読む前にオチが割れる
- Good: 「同値関係としての分類」「ループ不変条件と帰納法」
