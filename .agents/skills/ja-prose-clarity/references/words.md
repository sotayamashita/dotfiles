# 語・文レベルの書き換え規則

一文を取り出して判定できる問題。番号は SKILL.md の規則番号と同じ。

## 目次

1. 名詞化の過多 → 動詞で書く
2. 一文一義の違反 → 文を分ける
3. 語彙密度が高すぎる → 機能語で呼吸を入れる
4. 挿入補足の過多 → 削除テストで消すか、独立文にする
5. 漢語の過剰使用 → 和語を選ぶ
6. 専門用語の漏出 → 動作で書き下す
7. 固有名詞の過剰翻訳 → 定義元の公式表記に従う
8. カタカナ外来語の過剰 → 日本語で言い換える
9. 具体性の欠如 → 実際に何が起きるかで書く
10. 予測可能すぎる語彙選択 → その文章に固有の言葉を使う
11. 受身の過剰 → 行動の主体を主語にする
12. 近接反復 → 2回目以降を受けるか消す
13. 過剰な言い換え → 同じ語で通す
14. 構文テンプレートの反復 → 文の長さとリズムを変える
15. 根拠なきぼかし → 保険を消し、根拠のある不確実性だけ残す
16. 誇張表現 → 事実の範囲で書く
17. 二重否定 → 肯定形で書く
18. 冗長な言い回し → 意味を足さない語を消す
19. コロン直結の直訳構文 → 名詞で終えるか文で導入する
20. 組版記号の混入 → ASCII か日本語の記号に寄せる

## 1. 名詞化の過多 → 動詞で書く

名詞句を、主語と動詞のある文に戻す。

- Bad: 「知識の蓄積と探索の効率化」
- Good: 「知識を溜めて探しやすくする」

## 2. 一文一義の違反 → 文を分ける

「〜で、」「〜が、」「〜ので、」で複数の命題を1文に繋がない。一文は 50 字以内を目安にする。一文一義が優先で、文長の変化(規則14)は1命題の範囲で修飾の量を変えて作る。

判定: 一文を読み終えたとき、そこで意味が完結しているか。

- Bad: 「pi-autoresearch は AI コーディングエージェント pi 向けの拡張で、「試す → 測る → 残すか戻すか」を自動で繰り返す」
- Good: 「pi-autoresearch は AI コーディングエージェント pi の Extension。「試す → 測る → 残すか戻すか」を自動で繰り返す」

## 3. 語彙密度が高すぎる → 機能語で呼吸を入れる

助詞や接続詞を入れ、圧縮されていても話し言葉のように読める文にする。

- Bad: 「評価基準具体化・分離評価検証」
- Good: 「評価基準を具体化し、作る側と評価する側を分けて検証」

## 4. 挿入補足の過多 → 削除テストで消すか、独立文にする

1. 括弧の中を消して意味が変わらなければ消す(削除テスト)。
2. 削れない補足は括弧から出して独立した文にする。

- Bad: 「debug APK(develop flavor、API base URL: `https://api.example.com`)を BrowserStack で動かすと、API リクエストが全て HTTP 403 Forbidden(nginx 既定の 403 ページ)で弾かれる」
- Good: 「`app-debug.app` を BrowserStack で動かすと、API リクエストが全て HTTP 403 Forbidden で弾かれる」

## 5. 漢語の過剰使用 → 和語を選ぶ

専門用語ではない部分は和語で書く。

- 「結晶化した」→「はっきりした」「まとまった」
- 「効率化」→「楽にする」「速くする」
- 「蓄積」→「溜める」「残る」
- 「毀損する」→「削られる」
- 「我々」→「作り手」のような具体名詞

## 6. 専門用語の漏出 → 動作で書き下す

読み手が専門家でない文脈では、英語の専門用語を、それが意味する動作や状態に開く。

- Bad: 「rubric 駆動の evaluator-optimizer ループ」
- Good: 「品質基準に照らして自己採点し、足りなければ作り直す」
- Bad: 「Trace Store に append-only で蓄積」
- Good: 「やりとりの記録をそのまま全て残す」

## 7. 固有名詞の過剰翻訳 → 定義元の公式表記に従う

判定基準は「定義した側(企業、OSS コミュニティ、標準規格、API 仕様)が大文字・固有名詞としてその語を扱っているか」の一点。規則6と逆の判断になる。汎用的な専門用語は日本語に開き、公式に定義された固有名詞は原語のまま残す。

- Bad: 「pi 向けの拡張」(pi の `ExtensionAPI` で定義された正式役割名を訳している)
- Good: 「pi の Extension」

人名は原綴りで書く(Lehman)。歴史上の人物や定着した概念名は、通用しているカタカナの通称でよい。訳語は分野で慣用されている語を選ぶ(プッシュ通知は「配送」ではなく「配信」)。

## 8. カタカナ外来語の過剰 → 日本語で言い換える

外来語以外に適切な日本語があるなら日本語で書く。定着した語(バス、ガス、テレビ)と、Continuous Delivery のような専門用語・固有名詞(規則6・7で扱う)は対象外。迷ったら NINJAL の「外来語」言い換え提案を引く(2006年で更新が止まった資料なので、定着度は現在の用例で判断し直す)。

- Bad: 「スキーム」 Good: 「計画」
- Bad: 「(荷物の)デリバリー」 Good: 「配達」

## 9. 具体性の欠如 → 実際に何が起きるかで書く

抽象的なラベルを、読み手が場面を思い浮かべられる表現に置き換える。

- Bad: 「本番品質のLLMエージェント設計原則」
- Good: 「顧客に出せるLLMエージェントの作り方」
- Bad: 「プロアクティブなアラート」
- Good: 「実行時間が前週比30%以上伸びたテスト5件を月曜朝に一覧表示」
- Bad: 「知識を体に入れる」のような、指す内容が一意に決まらない比喩
- Good: 「身につく」と平易な動詞でそのまま言う

## 10. 予測可能すぎる語彙選択 → その文章に固有の言葉を使う

どの文章にも置ける「それっぽい言葉」を消し、その文章に実際に出てくる具体的な語彙を使う。

- Bad: 「包括的なテスト戦略を構築する」
- Good: 「ログイン・決済・通知の3経路を毎リリースで回すテストを書く」
- 空虚な動詞・決まり文句も同類: 「掘り下げる」「言語化する」「〜に他ならない」「正面から扱う」

## 11. 受身の過剰 → 行動の主体を主語にする

「行われます」「されます」を見たら、主体を主語に立てる。例外は、旧情報を文頭に置くために受身を使う場合(既知→未知の維持)。

- Bad: 「データの検証が行われます」
- Good: 「システムがデータを検証します」

## 12. 近接反復 → 2回目以降を受けるか消す

同じ語が見出し・初文・補足文の近い範囲で続くなら、「これ」「この仕組み」で受けるか、主語を省く。別の語への言い換えで逃げない(規則13の対)。

判定:

- 見出しに出た語を初文でも使う → 許容
- 初文の直後に同じ語がまた出る → 受けるか消す
- 対比や曖昧さの回避に必要 → 残す

例:

- Bad: 「本 ADR では AgenticSystem を…」の直後に「AgenticSystem 全体ではない」「SoftwareTesting 向けの AgenticSystem」と続く
- Good: 「AgenticSystem は…」の後は「判断対象は全体ではない」と受ける

## 13. 過剰な言い換え → 同じ語で通す

同一の対象は導入した語で呼び続ける(規則12の対)。

- Bad: 同じ文書で「ユーザー」「利用者」「エンドユーザー」が同じ人を指す
- Good: 文書全体で「ユーザー」で通す
- Bad: 「ユーザーがログインし、クライアントが設定を変更します」
- Good: 「ユーザーがログインし、設定を変更します」(直後なら主語を省く)

## 14. 構文テンプレートの反復 → 文の長さとリズムを変える

短い文と長い文を混ぜ、段落の終わり方を変える。避けるパターン:

- `It's not X — it's Y` 型の対句
- 全段落が「トピックセンテンス、サポート、まとめ」で均一
- 箇条書きの各項目が `**太字ラベル** — 説明` で始まる
- 太字の過剰使用、見出し全体の太字
- 絵文字を箇条書きの記号や強調に使う
- 形容や例をいつも3つ並べる
- 溜め(「ここには〜が潜んでいる」)や修辞疑問で導出を演出する

太字は初出の用語定義と論理の要所に限り、一節に1〜2箇所まで。導入済みの語への言及は「」で受ける。

## 15. 根拠なきぼかし → 保険を消し、根拠のある不確実性だけ残す

「かもしれません」「と思われます」「可能性があります」を見たら根拠を確認する。検証済みなら断定に直す。未検証なら「未検証だが」と理由ごと書く。出典は「専門家によれば」でぼかさず名指しする。逆方向にも注意する。推量を機械的に断定へ変えない。記録からの推定、読者が抱きそうな疑念、反実仮想を表す推量は保ち、断定に直せるのは本文内の根拠で命題が確定している場合に限る。

- Bad: 「この設定で改善する可能性があると思われます」
- Good: 「この設定で改善する」(検証済み)
- Good: 「未検証だが、この設定で改善するはずだ」(未検証)

## 16. 誇張表現 → 事実の範囲で書く

誇張語(「完全に」「究極の」「魔法のように」「パラダイムシフト」)は事実が支える範囲まで弱め、数字や条件で言えるなら置き換える。

- Bad: 「完全に問題を解決します」 Good: 「多くの問題を解決します」
- Bad: 「革命的な技術」 Good: 「ビルド時間を40%削った技術」

## 17. 二重否定 → 肯定形で書く

- Bad: 「できないわけではない」 Good: 「できる」(条件があるなら条件を書く)
- Bad: 「在留カード以外は必要ありません」 Good: 「在留カードを持ってきてください」

## 18. 冗長な言い回し → 意味を足さない語を消す

消して意味が変わらない語は消す。

- Bad: 「このAPIを使用することができます」 Good: 「このAPIを使用できます」
- Bad: 「調査を実施した」 Good: 「調査した」

## 19. コロン直結の直訳構文 → 名詞で終えるか文で導入する

判定: コロンの前が述語(動詞・形容詞・助動詞)なら不自然、名詞なら自然。

- Bad: 「実行します:」+コードブロック
- Good: 「例:」+箇条書き
- Good: 「実行方法は次の通りです。」+コードブロック

## 20. 組版記号の混入 → ASCII か日本語の記号に寄せる

英語圏の組版記号(`“”` `‘’` `—` `–` `…`)を持ち込まない。日本語として自然な記号(「」、。、・、())は使ってよい。不可視文字(ゼロ幅スペース、UTF8-MAC 濁点、制御文字)と半角カナは機械チェックに委ねる。

判定:

- 原文タイトルや引用 → `"` を使う
- コード、識別子、UI ラベル、仕様上の用語 → バッククォートを使う
- 日本語本文の強調や語としての言及 → 「」を使うか記号を外す
