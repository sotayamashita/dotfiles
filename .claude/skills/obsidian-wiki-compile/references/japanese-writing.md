# 日本語の書き方

LLM が生成する日本語には特有の読みづらさがある。Wiki 記事を書くときは以下のルールを守る。

詳細な根拠と研究は vault の `50-notes/LLMが生成する文章の読みづらさの正体と解決方法.md` を参照。

## ルール

### 動詞で書く（名詞化を避ける）

名詞句を主語＋動詞の文に戻す。誰が何をしたか見えるように書く。

- Bad: 「精度低下」「知識の蓄積と探索の効率化」
- Good: 「精度が落ちる」「知識を溜めて探しやすくする」

### 機能語で呼吸を入れる（語彙密度を下げる）

助詞、接続詞、つなぎの短い表現を入れる。圧縮された話し言葉のように読めるのが目標で、電報のようにしない。

- Bad: 「評価基準具体化・分離評価検証」
- Good: 「評価基準を具体化し、作る側と評価する側を分けて検証」

### 具体的に書く（抽象的なラベルを避ける）

読者が絵を思い浮かべられる表現に置き換える。

- Bad: 「本番品質のLLMエージェント設計原則」
- Good: 「顧客に出せるLLMエージェントの作り方」

### 和語を選ぶ（漢語の過剰使用を避ける）

技術用語はそのまま使ってよい。それ以外の周辺の言葉は和語で書く。声に出して不自然なら漢語が入りすぎている。

| 漢語（避ける） | 和語（使う） |
| --- | --- |
| 特権 | ありがたいもの |
| 展開する | 説明する、述べる |
| 効率化 | 楽にする、速くする |
| 蓄積 | 溜める、増やす |
| 達成 | できる、うまくいく |

### 文の長さとリズムを変える

同じ構文パターンの繰り返しを避ける。短い文と長い文を混ぜる。

### 予測可能すぎる語彙を避ける

「重要な」「包括的な」「革新的な」のような空虚な形容詞を避ける。その記事に実際に出てくる具体的な語彙を使う。

## リンクと説明の書き方

`[[link]] — description` のように1行にまとめない。リンクと説明は改行で分ける：

```markdown
- [[Context Anxiety]]
  - コンテキスト上限付近でモデルが急ぐ現象
```
